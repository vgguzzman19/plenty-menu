Descripción del Proyecto
Qué es y para qué sirve

Plenty es una carta digital con panel de administración para un restaurante de brunch en Platja d'Aro (Costa Brava). Los clientes escanean un código QR en la mesa y acceden a la carta desde su móvil, sin necesidad de instalar nada.

Funcionalidades principales
  • Carta digital pública — 29 productos en 7 categorías
  • Navegación por categorías con scroll suave (optimizada para móvil)
  • Selector de idioma: Español, English, Français — guardado en el navegador
  • Panel de administración protegido con contraseña
  • Gestión de productos y categorías desde el panel (crear, editar, eliminar)
  • Marcar productos como no disponibles en tiempo real
  • Base de datos en la nube (Supabase) — los cambios se ven al instante
  • Código QR generado automáticamente con la URL del restaurante
URL de producción La app está desplegada en Vercel y es accesible públicamente desde cualquier dispositivo con internet. Los clientes del restaurante acceden escaneando el QR de la mesa.

Flujo de datos

El usuario visita la web → page.tsx (servidor) llama a storage.ts → storage.ts consulta Supabase → los datos llegan al componente MenuClient.tsx (cliente) → el usuario ve la carta. El idioma se gestiona 100% en el cliente con localStorage.

4
Base de Datos — Supabase
Configuración de PostgreSQL en la nube
Paso 1 — Crear cuenta y proyecto

Ve a supabase.com → "New project" → elige nombre, región Europe (Frankfurt) y contraseña. Espera ~2 minutos.

Paso 2 — Crear las tablas (SQL Editor)

En el panel de Supabase: SQL Editor → New query. Pega y ejecuta:

-- TABLA DE USUARIOS ADMIN
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  username TEXT UNIQUE NOT NULL,
  password_hash TEXT NOT NULL,
  role TEXT NOT NULL DEFAULT 'admin'
);

-- TABLA DE CATEGORÍAS
CREATE TABLE categories (
  id SERIAL PRIMARY KEY,
  name TEXT NOT NULL,
  name_en TEXT,
  name_fr TEXT,
  emoji TEXT NOT NULL DEFAULT '🍽️',
  "order" INTEGER NOT NULL DEFAULT 0
);

-- TABLA DE PRODUCTOS
CREATE TABLE products (
  id SERIAL PRIMARY KEY,
  category_id INTEGER NOT NULL REFERENCES categories(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  description TEXT NOT NULL DEFAULT '',
  name_en TEXT,
  description_en TEXT,
  name_fr TEXT,
  description_fr TEXT,
  price DECIMAL(10,2) NOT NULL DEFAULT 0,
  image_url TEXT NOT NULL DEFAULT '',
  available BOOLEAN NOT NULL DEFAULT true,
  "order" INTEGER NOT NULL DEFAULT 0
);
TablaDescripciónColumnas clave
usersAdministradores del sistemausername, password_hash
categoriesSecciones de la cartaname, name_en, name_fr, emoji, order
productsPlatos y bebidasname, description, price, available, traducciones
Paso 3 — Obtener credenciales

En Supabase: Settings → API. Necesitas dos valores:

  • Project URL — algo como https://abcdef.supabase.co
  • service_role key — la clave secreta (NO la anon/public)
⚠️ Importante sobre la service_role key Esta clave tiene acceso total a tu base de datos. Nunca la pongas en código del frontend ni la subas a GitHub. Solo va en el archivo .env.local (local) y en las variables de entorno de Vercel.
Cómo se conecta el código a Supabase

El archivo src/lib/storage.ts crea un cliente de Supabase y exporta funciones para cada operación:

import { createClient } from "@supabase/supabase-js";

const supabase = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!
);

// Ejemplo: obtener todas las categorías ordenadas
export async function getCategories(): Promise<Category[]> {
  const { data, error } = await supabase
    .from("categories")
    .select("*")
    .order("order");

Archivo clave: src/lib/i18n.ts

Contiene el tipo Lang, los helpers para obtener el texto correcto y los textos de UI traducidos.

export type Lang = "es" | "en" | "fr";

// Devuelve el nombre del producto en el idioma activo
export function prodName(p: Product, lang: Lang): string {
  if (lang === "en" && p.name_en) return p.name_en;
  if (lang === "fr" && p.name_fr) return p.name_fr;
  return p.name; // español por defecto
}

// Textos de interfaz traducidos
export const ui = {
  es: { unavailable: "No disponible", empty: "Carta en preparación..." },
  en: { unavailable: "Not available", empty: "Menu in preparation..." },
  fr: { unavailable: "Non disponible", empty: "Menu en préparation..." },
};
Selector de idioma Aparece en la esquina superior derecha del hero (ES · EN · FR) en color dorado. Al cambiar de idioma, toda la carta se actualiza instantáneamente sin recargar la página.
6
Panel de Administración
Gestión de la carta en tiempo real
Acceso

URL: tudominio.com/admin → login con usuario y contraseña → sesión con JWT guardado en cookie httpOnly segura.

Qué puedes hacer desde el panel
  • Ver todos los productos y categorías
  • Crear nuevas categorías con emoji y orden
  • Añadir productos con nombre, descripción, precio e imagen
  • Editar cualquier producto o categoría
  • Marcar un producto como "No disponible" (se ve en la carta al instante)
  • Eliminar productos o categorías
  • Generar y descargar el código QR de la carta
Seguridad

El middleware de Next.js (middleware.ts) protege todas las rutas /admin/* y /api/* (excepto login). Sin token válido, redirige al login automáticamente.

Despliegue en Vercel
Cómo subir la app a producción
Subir código a GitHub

Crear repo en github.com, hacer git init, git add ., git commit y git push.

Conectar con Vercel

Ir a vercel.com → "New Project" → importar repo de GitHub → Vercel detecta Next.js automáticamente.

Añadir variables de entorno

Antes de hacer Deploy, en la sección "Environment Variables" añadir las 3 variables secretas (ver sección 9).

Deploy

Clic en "Deploy". En ~2 minutos la app está en producción con una URL pública.

Deploys automáticos

Cada vez que hagas git push a la rama main, Vercel redeploya automáticamente sin hacer nada más.

Deploy automático en cada push Una vez configurado, para actualizar la web basta con guardar los cambios en el código y hacer git push. Vercel lo detecta y publica la nueva versión en minutos.

Flujo para Futuros Proyectos
Cómo replicar esta arquitectura en otros proyectos
Crear el proyecto Next.js

Ejecutar npx create-next-app@latest nombre-proyecto --typescript --tailwind --app

Instalar Supabase

Ejecutar npm install @supabase/supabase-js en la carpeta del proyecto.

Crear proyecto en Supabase

supabase.com → New project → crear las tablas con SQL Editor → copiar URL y service_role key.

Crear .env.local

Poner las credenciales de Supabase y cualquier otro secreto en este archivo. Nunca subir a GitHub (añadir al .gitignore).

Crear src/lib/storage.ts

Inicializar el cliente de Supabase y exportar funciones async para cada operación (get, create, update, delete).

Construir la UI

Crear los componentes y páginas. Las páginas del servidor usan await para llamar a las funciones de storage.ts.

Subir a GitHub

Crear repo → git init → git add . → git commit -m "inicial" → git push.

Desplegar en Vercel

vercel.com → Import → añadir variables de entorno → Deploy. ¡Listo!

⚠️ .gitignore — archivos que NO deben subirse a GitHub Asegúrate de que el archivo .gitignore incluya: .env.local, node_modules/, .next/, data/

Variables de Entorno
Secretos necesarios para que la app funcione

Estas variables van en .env.local en local y en la configuración de Vercel en producción:

SUPABASE_URL=https://xxxxxxxxxxxx.supabase.co
SUPABASE_SERVICE_ROLE_KEY=eyJhbGci... (clave larga de Supabase)
JWT_SECRET=cualquier-cadena-larga-y-aleatoria-aqui
VariableDónde se obtienePara qué sirve
SUPABASE_URLSupabase → Settings → API → Project URLDirección de tu base de datos
SUPABASE_SERVICE_ROLE_KEYSupabase → Settings → API → Secret keysAcceso total a la BD desde el servidor
JWT_SECRETLo inventas tú (mín. 20 caracteres)Firma los tokens de sesión del admin
En Vercel estas variables se añaden en: Project Settings → Environment Variables. Sin ellas la app no puede conectarse a la base de datos y fallará.
